- Drift policy: when an item ships, **delete it** from this file. Provenance lives in the merge commit and release history.
- Status tag (2026-05-22 enterprise-readiness triage):
  - **T1** — release-blocking; absence allows lost user state, lost git history, or unrecoverable partial fleet state.
  - **T2** — contract integrity; absence allows the safe-by-default contract to drift or bypass.
  - **T3** — coverage, audit, or docs required before declaring the v1 enterprise release done.
  - **DEFER** — real value, but absence causes no safety harm at fleet-of-dozens scale.
  - **CUT** — considered and rejected; kept here so it doesn't reappear by accident. File a fresh item rather than reviving a cut entry verbatim if the reasoning changes.

- **§9.2 Rate limiting for fleet `gh` loops (`init fleet` repo creation, topics, roster) + docs.**
  - **Status:** OPEN (filed 2026-06-12; user directive: rate limit = **150**, unit unresolved — confirm before building). No rate-limit concept exists in the CLI today, so user docs must not mention one until this ships (implemented ≠ documented).
  - **Problem:** per-member fleet loops multiply `gh` API calls linearly with fleet size: `init fleet --create-repo-private|public -y` issues one `gh repo create` per selected directory (content-creating calls subject to GitHub's secondary limits, ~80 content-generating requests/min and 500/hr), `topics fleet`/the post-ship topics pass do per-repo `gh repo view` + delta `gh repo edit --add-topic` plus per-owner `gh repo list`, and `ship fleet` creates one GitHub release per member. Fine at fleet-of-dozens scale; at hundreds, content-creating bursts trip GitHub's secondary limits and surface as opaque 403s mid-apply, with no pacing and no retry guidance. (`git init` itself is local and exempt — in the init flow the limiter attaches to the accompanying `gh repo create` calls, not to git.)
  - **Open design question:** the unit for 150 — 150/min exceeds GitHub's own content-creation secondary limit (~80/min), 150/hr is conservative, 150-per-run is a cap rather than a rate. Resolve with the maintainer before implementation.
  - **Shape (recommendation):** ONE self-describing key ([[feedback_config_ux]]), e.g. `github.rate_limit: 150`, enforced as inter-call pacing in the shared `gh` helper so every loop inherits it; env override `MANIFEST_CLI_GH_RATE_LIMIT` (prefix rule); default unset = current unthrottled behavior. Pacing only — never a consent surface and never a partial-run filter (no member skipping; [[feedback_fleet_consent_model]]); preview runs already make zero `gh` calls, so only apply mode is paced; throttling must be visible (one pacing notice, never silent — [[feedback_surface_defects]]). Capped-and-reported precedent: `MANIFEST_CLI_FLEET_TOPICS_ROSTER_LIMIT`.
  - **Docs (the original ask):** when it ships, document the key and the affected flows (`init fleet` Phase 2 repo creation, `topics fleet` + post-ship pass, `ship fleet` releases) in COMMAND_REFERENCE + USER_GUIDE in the same change — not before.
  - **Anchor:** [`modules/core/manifest-shared-functions.sh`](../modules/core/manifest-shared-functions.sh) (`gh repo create` helper, `_manifest_require_gh`), [`modules/fleet/manifest-fleet.sh`](../modules/fleet/manifest-fleet.sh) (Phase 2 bootstrap loop), [`modules/fleet/manifest-fleet-topics.sh`](../modules/fleet/manifest-fleet-topics.sh) (view/edit/list loops), [`modules/core/manifest-yaml.sh`](../modules/core/manifest-yaml.sh) (key registration).

- **§9.3 GitHub owner/organization targeting for repo creation.**
  - **Status:** OPEN (filed 2026-06-12, user ask). Today `_manifest_gh_repo_create` passes the bare basename to `gh repo create`, so every repo lands under the **authenticated personal account** — there is no way to create under an organization, even though the helper's own failure hint mentions "no permission for org". `gh` has no default-owner setting of its own (`gh repo create <name>` always defaults to the authenticated user); targeting an org requires the `<owner>/<name>` argument form, so Manifest needs its own knob.
  - **Shape (recommendation):** ONE self-describing key ([[feedback_config_ux]]), e.g. `github.owner: <org-or-user>`; default unset = authenticated user (today's behavior, zero migration). Registered in `_MANIFEST_YAML_TO_ENV` (the `topics.from_name` pattern) so it works fleet-wide via yaml and host-locally via `config set --layer global`; env override `MANIFEST_CLI_GITHUB_OWNER` (prefix rule). When set, the helper creates `<owner>/<name>` — one change site covers all three flows (`init repo`, `prep`, `init fleet` Phase 2 `--create-repo-*`).
  - **Consent requirement:** the preview line must show the full target slug (`would gh repo create: <owner>/<name> (private)`) so the destination owner is visible BEFORE `-y` — creating dozens of repos under the wrong owner is exactly what the preview exists to catch. Fail loud on create (gh's permission error already surfaces, per-directory rc 1 classification exists); no silent fallback to the personal account on org failure.
  - **Mixed-owner fleets (one fleet, repos destined for different owners) — user-raised 2026-06-12.** Resolution order, cheapest first:
    1. **Pre-existing `origin` wins (already built, no new code).** `_manifest_gh_repo_create` skips any directory that already has an `origin` remote ("origin already configured — skipping gh repo create"). So the few exceptions in an otherwise-uniform fleet are handled by creating those remotes first by any means (under any owner); the fleet-wide `github.owner` covers the rest, and the pre-remoted ones are left untouched. This is the recommended answer for "most repos → org A, a handful → elsewhere" and needs nothing beyond §9.3's single key.
    2. **First-class per-member owner = an `OWNER` column in `manifest.fleet.tsv`** — promote to real work ONLY if a fleet genuinely split across owners at scale appears (where hand-creating half the remotes defeats `--create-repo-*`). **Cost is not additive:** the TSV is a fixed 7-column schema (`SELECT NAME PATH TYPE HAS_GIT REMOTE_URL BRANCH`) and `merge_start_tsv` re-emits exactly those 7 fields on every re-scan, **silently dropping any extra column** (same wipe class as the topics remote-only-row hazard, via columns). An `OWNER` column therefore requires extending the canonical schema in lockstep — header, every 7-field `IFS=$'\t' read` consumer, `merge_start_tsv`'s preserve-map (add `existing_owners`), and `generate_start_tsv` — and the preview must show each row's resolved `<owner>/<name>`. Deliberate schema change, gated behind a real ask.
  - **Deferred / rejected (do not pre-build):** a `--owner` per-invocation flag; a **per-service owner map in the yaml** (`services.<x>.github.owner`) — same override-map shape already deferred for topics and rejected in §6.1 ([[feedback_config_ux]]: one self-describing field, not composed per-service maps); org *validation* preflight (`gh api orgs/<org>`) beyond gh's own create-time error.
  - **Anchor:** [`modules/core/manifest-shared-functions.sh`](../modules/core/manifest-shared-functions.sh) (`_manifest_gh_repo_create`, pre-existing-remote skip), [`modules/core/manifest-yaml.sh`](../modules/core/manifest-yaml.sh) (key registration), [`modules/fleet/manifest-fleet-detect.sh`](../modules/fleet/manifest-fleet-detect.sh) (`merge_start_tsv`, `generate_start_tsv` — the 7-column schema for an OWNER column), [`modules/core/manifest-init.sh`](../modules/core/manifest-init.sh) + [`modules/core/manifest-prep.sh`](../modules/core/manifest-prep.sh) + [`modules/fleet/manifest-fleet.sh`](../modules/fleet/manifest-fleet.sh) (preview lines). Related: §9.2 (rate limiting paces these same creation loops).

- **§9.4 Fleet-root coordination repo: local git-init + allowlist `.gitignore` (`init fleet`).**
  - **Status:** IMPLEMENTED, pending release (2026-06-14, user ask). Motivated by a real incident: a date-versioned fleet (`fidence`) had no git repo at its root, so `manifest ship fleet` (even `--local`) tripped the root-git requirement; a hand-made `git init` + `git add -A` then committed+pushed 188 non-coordination files (incl. a `secure/` .NET auth-service source tree). Manifest git-inits **member** dirs but never the **fleet root** (`_fleet_init_directory`), so every developer of a date-versioned fleet hits the same wall.
  - **Shape (implemented):** new `create_fleet_gitignore()` writes an **allowlist** `.gitignore` at the fleet root — `/*` then re-include only `manifest.fleet.config.yaml`, `manifest.fleet.tsv`, `FLEET_VERSION`, `CHANGELOG_FLEET.md`, `.gitignore`. Member repos, service source, secrets, and host-local `manifest.config.local.yaml` can never be tracked; the layout stays in `manifest.fleet.tsv` (structure-of-record). No-clobber mirrors `ensure_gitignore_smart()` (existing populated `.gitignore` preserved; allowlist written to `.gitignore.manifest`). `_fleet_init()` also ensures the root is a **local-only** git repo (`git init`, no remote) for **all** fleet roots (user decision), shown in the Phase-2 **preview** and applied on `-y` (consent-model C). Allowlist is static → `fleet_update` needs no member-sync site.
  - **Atomic + idempotent + fleet-root versioning (follow-up, implemented 2026-06-14):** root detection now uses the dir's *own* `.git` (`_fleet_dir_is_own_git_repo` → `manifest_discovery_is_git_repository`), not `--is-inside-work-tree` (which falsely passed when merely nested in a parent — now warns + still creates a separate coordination repo); the allowlist write is **atomic** (temp + `mv -f`); re-runs are **idempotent** (an existing allowlist is a clean no-op — no redundant `.gitignore.manifest`); a failed root `git init` now **fails the command**. **`ship fleet` now bumps + commits `FLEET_VERSION` at the root** after members ship (`_fleet_next_version` for date/semver/increment/none; `_fleet_root_release`). **SECURITY:** the root commit stages coordination files **by name** and verifies the staged set — never `git add .` (which `commit_changes` uses) — aborting if any non-coordination path is staged; honors `--local` (commit, no push) and preview/`--dry-run`; idempotent (skips when scheme=none, no member shipped, or version unchanged).
  - **Anchor:** [`modules/fleet/manifest-fleet.sh`](../modules/fleet/manifest-fleet.sh) (`create_fleet_gitignore`/`_fleet_write_allowlist_gitignore`/`_fleet_dir_is_own_git_repo`, `_fleet_init` root step, `_fleet_next_version`, `_fleet_root_release`, `fleet_ship` wiring), [`modules/core/manifest-init.sh`](../modules/core/manifest-init.sh) (`_manifest_init_fleet_dry_run_phase2` preview), tests [`tests/fleet_root_gitignore.bats`](../tests/fleet_root_gitignore.bats) + [`tests/fleet_root_version.bats`](../tests/fleet_root_version.bats). Deferred: unified `CHANGELOG_FLEET.md` generation; fleet git tag; custom (non-`FLEET_VERSION`) version-file names in the static allowlist. ## §8 — Enterprise-readiness audit (2026-06-05)

Seven-dimension code audit (atomicity, concurrency, security, test-gating, audit/observability, config/portability, ops/governance). The top tier — §8.1b/§8.1f, §8.2a, §8.3a/b/d, §8.4a/b, consent-model **C**, and the `manifest first` completion — **shipped in CLI 52.4.1** (see CHANGELOG / release history) and is removed here per the drift policy. Open items below.

- **§8.1 Ship transaction — post-tag integrity (remaining).**
  - **§8.1a** [T1, conditional on ≥2 remotes] Multi-remote partial push mis-classified as rollback-safe: `push_changes` returns on the first failing remote, so a branch/tag already public on remote #1 can be offered a `git reset --hard` recovery; `resume` only probes `origin`. Track per-remote outcome (`push_status="partial"`), suppress hard-reset advice, emit per-remote retry. Anchor: [`modules/git/manifest-git.sh`](../modules/git/manifest-git.sh) (`push_changes`), [`modules/workflow/manifest-orchestrator.sh`](../modules/workflow/manifest-orchestrator.sh) (`emit_ship_failure_report`, resume probe).
  - **§8.1d** [T2] `resume` re-derives the formula sha256 from the current tarball with no compare against the existing formula value → silent drift. Read existing `sha256`, warn + require `--force` on mismatch. (Pairs with §8.1g.)
  - **§8.1e** [T2] Tap `git pull --ff-only` failure is swallowed ("continuing with local state") → tap formula committed onto a stale/diverged tap. Treat ff-only failure as fatal for the tap sync. Anchor: [`modules/core/manifest-core.sh`](../modules/core/manifest-core.sh) (`update_homebrew_formula`). *NB: distinct from the "workspace tap clone 1-behind after every ship" ordering bug — that was fixed in v54.0.1 (commits a094d5c+6861b0a); don't conflate the two.*
  - **§8.1g** [T3] No durable transaction-state file; recovery is inferred from git/gh/brew probes. Journal `{version, tag, commit_sha, intended_sha256, last_step}` per step; have resume prefer it.
