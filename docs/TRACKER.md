# Manifest CLI — Tracker

Open work for the Manifest CLI repo, as one flat list.

- Every item names a concrete deliverable and an anchor; the `§X.Y` label is a stable ID (referenced from commits, memory, and the Cloud/workspace trackers — do not renumber).
- Drift policy: when an item ships, **delete it** from this file. Provenance lives in the merge commit and release history.
- Status tag (2026-05-22 enterprise-readiness triage):
  - **T1** — release-blocking; absence allows lost user state, lost git history, or unrecoverable partial fleet state.
  - **T2** — contract integrity; absence allows the safe-by-default contract to drift or bypass.
  - **T3** — coverage, audit, or docs required before declaring the v1 enterprise release done.
  - **DEFER** — real value, but absence causes no safety harm at fleet-of-dozens scale.
  - **CUT** — considered and rejected; kept here so it doesn't reappear by accident. File a fresh item rather than reviving a cut entry verbatim if the reasoning changes.

- **§9.1 GitHub topics projection from repo-name slugs (fleet).**
  - **Status:** SHIPPED — Phases 1+2 in v53.3.0, 2026-06-12 additions below in v53.4.0 (filed 2026-06-11). **Not a canonical change** — feature for third-party fleets with slug-convention repo names; the fidenceio fleet yaml never sets the key (the maintainer enables it host-locally via the global config layer).
  - **2026-06-12 additions (explicit user ask, reversing the earlier subcommand rejection):** (a) `manifest topics fleet [-y|--dry-run]` — direct entry to the same single hook, preview default, its own apply hint; fleet-scoped, no git repo required (core dispatch + completions + help + COMMAND_REFERENCE in the same change). (b) **Quiet post-ship pass:** a completed `ship fleet -y` (not `--local`) runs the projection in apply mode via `_fleet_ship_topics_pass` — silent when up-to-date/degraded, one `🏷️` summary line when pushed or failed, never fails a completed ship, roster suppressed. (c) **Host-local enablement:** `topics.from_name` registered in `_MANIFEST_YAML_TO_ENV` → `manifest config set topics.from_name inner --layer global` turns topics on for one machine; the layered/env value takes precedence over the fleet yaml. A shell-profile `export` was tried and rejected — it leaks into every process env and breaks the suite's clean-env assumption.
  - **Scope (Phase 1):** ONE new YAML leaf key in `manifest.fleet.config.yaml` — `topics.from_name: inner | all | all-but-first` ([[feedback_config_ux]]: one self-describing field). Derivation: split the GitHub repo name (origin-URL slug, not the local dir name) on dots; `inner` drops first+last slug — `fidence.service.accounting.avalara` → `service`, `accounting`. Apply at `update fleet`: **read existing topics first** (`gh repo view --json repositoryTopics`), push only the delta via `gh repo edit --add-topic` — already-defined topics are never re-pushed; unchanged repos cost zero writes. **Additive-only for ALL topics, including `fleet-<name>`** — a repo can belong to two fleets (two roots), so removing a "stale" `fleet-*` topic from one fleet's run would fight the other; removal stays manual (deferred with Phase 2). Preview block = notice, `-y` = consent (consent-model C). `gh` missing/unauth/non-GitHub origin → skip rc 2 (GitHub-release precedent).
  - **Empty/null contract:** absent block, empty/null value, and zero-derived-slug repos are ALL silent complete no-ops (zero `gh` calls). Configured→emptied = stop managing, never sweep. Present-but-invalid value (`from_name: midle`) = fail loud (wire into §8.4c when it lands).
  - **Phase 2 roster — IMPLEMENTED (read-only, report-only):** with topics on, the run also enumerates the **full** org list per enrolled owner (`gh repo list <owner> --no-archived --json name`; the topic-filtered query alone can't bootstrap or find untagged new repos), candidate-matches by the SAME `from_name` dot-slug convention (family = shared first slug with an enrolled member — one convention, two uses, no new config), and **reports** unenrolled family repos with a clone-to-enroll hint. Candidates are deliberately NOT written to the TSV (`merge_start_tsv` rebuilds the TSV from the local scan, so a remote-only row is silently wiped on the next apply) and NEVER topic-stamped (writes to un-enrolled repos break the consent boundary; a raw org-wide tag loop is a transparent `gh` one-liner, not a Manifest mode) — cloning into the fleet root IS the enrollment act. Failed org listing = per-owner notice, never fatal; `MANIFEST_CLI_FLEET_TOPICS_ROSTER_LIMIT` (default 1000) caps the listing and hitting the cap is reported, never silent. Distinct from rejected §6.2, which was *local* drift.
  - **Deferred (do not pre-build):** per-service `topics:` override list; single-repo `project.topics` (`gh repo edit` is already a one-liner); any topic removal (multi-fleet membership makes attribution unsafe); TSV row injection for remote-only candidates (revisit only if merge semantics ever become path-independent).
  - **Rejected within scope:** `--topic` run-filter flags (= `--only` in costume, consent model). (The earlier `manifest topics` subcommand rejection was reversed by explicit user request 2026-06-12 — see additions above.)

- **§9.2 Rate limiting for fleet `gh` loops (`init fleet` repo creation, topics, roster) + docs.**
  - **Status:** OPEN (filed 2026-06-12; user directive: rate limit = **150**, unit unresolved — confirm before building). No rate-limit concept exists in the CLI today, so user docs must not mention one until this ships (implemented ≠ documented).
  - **Problem:** per-member fleet loops multiply `gh` API calls linearly with fleet size: `init fleet --create-repo-private|public -y` issues one `gh repo create` per selected directory (content-creating calls subject to GitHub's secondary limits, ~80 content-generating requests/min and 500/hr), `topics fleet`/the post-ship topics pass do per-repo `gh repo view` + delta `gh repo edit --add-topic` plus per-owner `gh repo list`, and `ship fleet` creates one GitHub release per member. Fine at fleet-of-dozens scale; at hundreds, content-creating bursts trip GitHub's secondary limits and surface as opaque 403s mid-apply, with no pacing and no retry guidance. (`git init` itself is local and exempt — in the init flow the limiter attaches to the accompanying `gh repo create` calls, not to git.)
  - **Open design question:** the unit for 150 — 150/min exceeds GitHub's own content-creation secondary limit (~80/min), 150/hr is conservative, 150-per-run is a cap rather than a rate. Resolve with the maintainer before implementation.
  - **Shape (recommendation):** ONE self-describing key ([[feedback_config_ux]]), e.g. `github.rate_limit: 150`, enforced as inter-call pacing in the shared `gh` helper so every loop inherits it; env override `MANIFEST_CLI_GH_RATE_LIMIT` (prefix rule); default unset = current unthrottled behavior. Pacing only — never a consent surface and never a partial-run filter (no member skipping; [[feedback_fleet_consent_model]]); preview runs already make zero `gh` calls, so only apply mode is paced; throttling must be visible (one pacing notice, never silent — [[feedback_surface_defects]]). Capped-and-reported precedent: `MANIFEST_CLI_FLEET_TOPICS_ROSTER_LIMIT`.
  - **Docs (the original ask):** when it ships, document the key and the affected flows (`init fleet` Phase 2 repo creation, `topics fleet` + post-ship pass, `ship fleet` releases) in COMMAND_REFERENCE + USER_GUIDE in the same change — not before.
  - **Anchor:** [`modules/core/manifest-shared-functions.sh`](../modules/core/manifest-shared-functions.sh) (`gh repo create` helper, `_manifest_require_gh`), [`modules/fleet/manifest-fleet.sh`](../modules/fleet/manifest-fleet.sh) (Phase 2 bootstrap loop), [`modules/fleet/manifest-fleet-topics.sh`](../modules/fleet/manifest-fleet-topics.sh) (view/edit/list loops), [`modules/core/manifest-yaml.sh`](../modules/core/manifest-yaml.sh) (key registration).

- **§9.3 GitHub owner/organization targeting for repo creation.**
  - **Status:** OPEN (filed 2026-06-12, user ask). Today `_manifest_gh_repo_create` passes the bare basename to `gh repo create`, so every repo lands under the **authenticated personal account** — there is no way to create under an organization, even though the helper's own failure hint mentions "no permission for org". `gh` has no default-owner setting of its own (`gh repo create <name>` always defaults to the authenticated user); targeting an org requires the `<owner>/<name>` argument form, so Manifest needs its own knob.
  - **Shape (recommendation):** ONE self-describing key ([[feedback_config_ux]]), e.g. `github.owner: <org-or-user>`; default unset = authenticated user (today's behavior, zero migration). Registered in `_MANIFEST_YAML_TO_ENV` (the `topics.from_name` pattern) so it works fleet-wide via yaml and host-locally via `config set --layer global`; env override `MANIFEST_CLI_GITHUB_OWNER` (prefix rule). When set, the helper creates `<owner>/<name>` — one change site covers all three flows (`init repo`, `prep`, `init fleet` Phase 2 `--create-repo-*`).
  - **Consent requirement:** the preview line must show the full target slug (`would gh repo create: <owner>/<name> (private)`) so the destination owner is visible BEFORE `-y` — creating dozens of repos under the wrong owner is exactly what the preview exists to catch. Fail loud on create (gh's permission error already surfaces, per-directory rc 1 classification exists); no silent fallback to the personal account on org failure.
  - **Deferred (do not pre-build):** a `--owner` per-invocation flag; per-member owner overrides in the TSV/yaml; org *validation* preflight (`gh api orgs/<org>`) beyond gh's own create-time error.
  - **Anchor:** [`modules/core/manifest-shared-functions.sh`](../modules/core/manifest-shared-functions.sh) (`_manifest_gh_repo_create`), [`modules/core/manifest-yaml.sh`](../modules/core/manifest-yaml.sh) (key registration), [`modules/core/manifest-init.sh`](../modules/core/manifest-init.sh) + [`modules/core/manifest-prep.sh`](../modules/core/manifest-prep.sh) + [`modules/fleet/manifest-fleet.sh`](../modules/fleet/manifest-fleet.sh) (preview lines). Related: §9.2 (rate limiting paces these same creation loops).

Track per-remote outcome (`push_status="partial"`), suppress hard-reset advice, emit per-remote retry. Anchor: [`modules/git/manifest-git.sh`](../modules/git/manifest-git.sh) (`push_changes`), [`modules/workflow/manifest-orchestrator.sh`](../modules/workflow/manifest-orchestrator.sh) (`emit_ship_failure_report`, resume probe).
